# EvoSkill 自进化 Agent 框架 -- 技术全景文档

## 一句话总结

EvoSkill 是一个让 AI Agent **自己学会变强**的框架：给它一个目标（提高点击率 / 答对更多题），它会自动尝试不同策略、评估效果、淘汰弱者、进化出更好的策略，无需人工干预。

---

## 这个项目是什么

### 解决什么问题

传统 AI Agent 的 prompt 是人写的，写完就固定了。但现实世界在变化——搜索引擎算法在更新、用户偏好在变、竞争对手在优化。**人工调 prompt 跟不上变化速度**。

EvoSkill 让 prompt 策略自动进化：
1. 生成多种策略（skills）
2. 在真实环境中测试
3. 用数据淘汰差的、保留好的
4. 基于失败分析提出新策略
5. 循环往复，越来越强

### 应用在两个领域

| 领域 | 目标 | 反馈信号 | 状态 |
|------|------|---------|------|
| **SEO CTR 优化** | 提高 Google 搜索结果的点击率 | Google Search Console 真实 CTR 数据 | 开发完成，数据太稀疏未大规模验证 |
| **OfficeQA 文档推理** | 答对尽可能多的金融文档问答题 | 答案正确率（1% 容差模糊匹配） | 竞赛进行中，当前排名 #7（152 分） |

### 目标用户

- **SEO 从业者**：想自动优化 title/description 但不想天天盯着数据
- **AI 竞赛参与者**：想让 agent 通过数据驱动自动进化 prompt 策略
- **AI 研究者**：研究 "skill-based agent evolution" 方向

## 整体架构：三层设计

```
┌─────────────────────────────────────────────────┐
│           提交层 (officeqa/)                      │
│  竞赛入口，只有配置文件和运行时 skills            │
│  arena.yaml + skills/*/SKILL.md                  │
└──────────────────┬──────────────────────────────┘
                   │ 进化产出的 skills 同步到这里
┌──────────────────▼──────────────────────────────┐
│           领域层 (seo_agent/ 或 office_qa/)       │
│  每个领域独立实现：数据获取、生成、评估、记录      │
│  data_client + executor + evaluator + intervention│
└──────────────────┬──────────────────────────────┘
                   │ 评估结果喂给进化引擎
┌──────────────────▼──────────────────────────────┐
│           引擎层 (engine/)                        │
│  跨领域复用：Pareto 前沿 + 进化记忆 + 策略提议    │
│  frontier.py + memory.py + proposer.py            │
└─────────────────────────────────────────────────┘
```

**核心思想**：引擎层不关心你在优化什么（SEO 还是 QA），它只关心"哪些策略好、哪些差、下一个该试什么"。领域层负责具体的"怎么生成、怎么评估"。

## 进化循环：怎么运作的

每一步进化包含 5 个阶段：

```
    ┌──────────┐
    │ 1. 生成   │ 用当前 skills 生成结果（标题/答案）
    └────┬─────┘
         │
    ┌────▼─────┐
    │ 2. 评估   │ 用真实数据或 LLM-as-Judge 打分
    └────┬─────┘
         │
    ┌────▼─────┐
    │ 3. 淘汰   │ Pareto 前沿淘汰弱 skills
    └────┬─────┘
         │
    ┌────▼─────┐
    │ 4. 分析   │ LLM 分析失败原因，提出新方向
    └────┬─────┘
         │
    ┌────▼─────┐
    │ 5. 进化   │ 生成新 SKILL.md，加入前沿
    └──────────┘
         │
         └──→ 回到第 1 步
```

### 阶段详解

#### 1. 生成

Agent 从当前活跃的 skills 中轮流选一个，用它来生成结果：
- SEO：给定一个页面 URL + 搜索关键词，用 skill 策略生成新的 title 和 description
- QA：给定一个金融问题 + 语料库，用 skill 策略推理出答案

Skills 是什么？就是一段 Markdown 指令，告诉 LLM "用什么思路来做这件事"。比如：
- `curiosity_gap`：用好奇心缺口技巧写标题（"你不知道的 3 个原因..."）
- `multi_step_calculation`：遇到需要多步计算的问题，先分解步骤再逐步算

#### 2. 评估

两种评估方式：

**真实数据评估**（首选）：
- SEO：等 1 天后看 Google 实际点击率变化（CTR lift），做统计检验
- QA：和标准答案比对（1% 容差模糊匹配）

**LLM-as-Judge 代理评估**（备选）：
- 当真实数据太少时（比如网站每天只有 2 次展示），用另一个 LLM 打分
- 5 个维度各 0-10 分，总分映射到 [-1, 1] 区间
- 好处：秒出结果，不用等数据积累
- 坏处：代理分数不等于真实效果

#### 3. 淘汰（Pareto 前沿）

维护最多 15 个 skills 的"精英池"。用三维指标评判：
- **平均提升幅度**（CTR lift 或正确率）：权重 50%
- **胜率**（成功次数/总次数）：权重 30%
- **覆盖度**（评估过多少次）：权重 20%

淘汰规则：
- **Pareto 支配**：如果 skill A 在所有维度都 >= skill B，且至少一个维度 > B，则 B 被淘汰
- **K-精英**：如果池子超过 15 个，按加权综合分数排序，保留 top 15
- **新手保护**：覆盖度=0 的新 skill 免淘汰（至少让它被测试一次）

#### 4. 分析

用 Claude Sonnet（强推理模型）做两件事：

**失败分析**：把所有失败/不确定的评估结果喂给 LLM，问它"你觉得为什么这些策略没用？有什么共同模式？"

**策略提议**：基于失败分析 + 进化记忆（什么有效、什么无效），提出一个新策略方向。比如："之前试过好奇心缺口但对品牌词无效，建议改用直接价值确认策略"。

#### 5. 进化

用 Claude Sonnet 根据策略提议，生成一个全新的 SKILL.md 文件，保存到 `skills/{name}/v{N}/SKILL.md`。自动版本递增，方便回溯。

## 进化记忆系统

进化引擎有"长期记忆"，跨步骤传递经验：

```json
{
  "promising_directions": [
    {"direction": "品牌词用直接价值确认", "evidence": "10 次评估，80% 胜率"}
  ],
  "failed_directions": [
    {"direction": "好奇心缺口用于品牌词", "evidence": "10 次评估，0% 提升"}
  ],
  "effective_patterns": [
    {"pattern": "标题开头放搜索关键词", "source_skills": ["intent_match_confirmation"]}
  ]
}
```

每类最多 10 条，FIFO 淘汰最旧的。这些记忆会注入到分析和提议的 LLM prompt 中，避免重复犯错。

## SEO 领域：具体怎么做

### 数据流

```
Google Search Console API
  → 按天缓存到 data/gsc/{date}.parquet
  → 聚合成 DataFrame
  → 找低 CTR 页面（opportunity_score = 曝光量 × CTR 差距）
  → 用 skill 生成新 title/description
  → 记录到 data/interventions.jsonl
  → [等 1 天]
  → 评估 CTR 变化（或 LLM 打分）
  → 喂给进化引擎
```

### 关键模块

| 模块 | 作用 | 一句话解释 |
|------|------|----------|
| `gsc_client.py` | 数据源 | 从 Google Search Console 拉点击/曝光数据 |
| `opportunity.py` | 机会识别 | 找出"曝光高但点击率低于基准"的页面 |
| `executor.py` | 内容生成 | 用 skill 策略让 LLM 生成新 title/description |
| `evaluator.py` | 真实评估 | 比较修改前后的 CTR，做统计检验 |
| `llm_evaluator.py` | 代理评估 | 数据不够时用 LLM 打分代替 |
| `intervention.py` | 审计日志 | 记录每次修改（什么时候、改了什么、效果如何） |

### 遇到的问题

我们的测试站点 meetspot 每天只有约 2 次搜索曝光。要做一次有统计意义的 A/B 测试至少需要 30 次曝光——也就是要等 15 天。这让真实 CTR 评估几乎不可能。

**解决方案**：LLM-as-Judge。不等真实数据，立刻用 AI 打分。虽然代理分数不等于真实效果，但它让进化循环能跑起来，产出了 18 个 evolved skills。

## OfficeQA 领域：竞赛怎么做

### 竞赛规则

- **赛题**：246 道基于美国财政部月报（Treasury Bulletin）的金融问答题
- **评估**：MiniMax M2.5 模型运行你的 agent，在 Docker 容器中回答问题
- **评分**：`Score = 正确题数 × (1 + 成本调整 + 延迟调整)`
- **限制**：每天 3 次提交，禁止预计算答案
- **截止**：2026 年 4 月 4 日

### Agent 在容器里怎么工作

```
Docker 容器启动
  → 加载 skills（6 个 SKILL.md 文件）
  → 对每道题：
    1. 解析问题（什么数据？什么年份？什么计算？什么格式？）
    2. 搜索语料库（grep + 同义词回退）
    3. Python 解析表格（split by |，计数列索引）
    4. Python 计算（formulas.py 公式库）
    5. 验证答案合理性
    6. 写入 /app/answer.txt
```

### 当前 6 个 Skills

| Skill | 作用 |
|-------|------|
| `00_system_instructions` | 核心工作流 + 内嵌 formulas.py（11 个金融公式） |
| `01_corpus_guide` | 文件命名规则 + 日期→文件映射 + 6 步搜索策略 |
| `02_table_parsing` | 表格解析技巧 + 列识别 + 特殊值处理 |
| `03_formula_library` | 公式库使用指南（percent_change, CAGR, HP filter 等） |
| `04_domain_knowledge` | 财政年规则 + 历史事件日期 + 金本位国家 |
| `05_answer_format` | 输出格式规范 + 评分逻辑对齐规则 |

### 评分逻辑（reward.py）的关键规则

这些直接影响我们的 skill 设计：

1. **Base number 比较**：`"543 million"` 的 base=543。你的答案写 `"543"` 就能匹配，但写 `"543000000"` 反而不匹配
2. **年份过滤**：答案中 1900-2100 之间的整数会被当年份忽略（除非正确答案本身就是年份）
3. **多数字全匹配**：列表答案要求所有数字都匹配，少一个就 0 分
4. **文本重叠**：日期答案（如 "March 1977"）要求文字和数字都匹配

## 关键设计决策和原因

### 为什么用 Markdown 存 Skills？

而不是 JSON/YAML/Python？

- **人能读**：非工程师也能看懂策略内容
- **Git diff 友好**：v1 到 v2 改了什么一目了然
- **LLM 友好**：Markdown 是 LLM 最擅长生成的格式
- **可回滚**：版本目录结构 `v1/`、`v2/`，随时切回旧版

### 为什么每个模块各有 `_get_client()`？

OpenRouter client 的创建代码在每个模块里重复了。这是**有意为之**——不抽象成共享 helper，因为：
- 各模块可能需要不同的超时/重试配置
- 减少模块间耦合
- 三行重复代码 < 一个过早抽象

### 为什么 intervention.jsonl 是 append-only？

用追加写入而不是更新记录：
- **崩溃安全**：追加写入不会破坏已有数据
- **完整审计**：能看到每次修改的历史
- **简单可靠**：不需要数据库，不需要事务

读取时用 last-write-wins 去重：同一个 ID 的最后一条记录就是当前状态。

### 为什么用原子写入（tmpfile + rename）？

Parquet 文件和 checkpoint 都用临时文件写入后重命名：
- 写入过程中崩溃？临时文件不影响原文件
- 重命名是操作系统级原子操作，不会出现写一半的状态
